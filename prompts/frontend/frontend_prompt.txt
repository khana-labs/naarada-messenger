# Narada Messenger — Complete Frontend and Dashboard Development Prompt

Build a polished, responsive frontend prototype for **Narada Messenger**, a decentralized emergency communication platform designed to operate when the internet, cellular towers, cloud servers, and conventional messaging systems are unavailable.

Narada Messenger transforms nearby smartphones into temporary communication nodes. Each device can discover other devices, securely store encrypted messages, carry them physically, and forward them toward the receiver’s last-known location.

The website is a hackathon simulation of this system. It must visually demonstrate messaging, nearby-device discovery, geographic routing, emergency alerts, message relaying, network health, and authority response.

## 1. Main Product Goal

Design the interface for people operating under extreme stress during disasters, network blackouts, wars, infrastructure failures, or other emergencies.

The interface must be:

* Immediately understandable
* Highly readable
* Minimal
* High contrast
* Fast to navigate
* Mobile-friendly
* Usable with one hand
* Battery-conscious
* Accessible during panic
* Free from unnecessary decoration
* Clear even at low screen brightness

The website must look futuristic and emergency-oriented, but **must not sacrifice readability for visual effects**.

The user should understand the most important information within three seconds:

1. Is the Narada network active?
2. Is the user safe or in danger?
3. Are nearby relay devices available?
4. Are messages pending, relaying, or delivered?
5. How can the user send a message or SOS alert immediately?

---

# 2. Core Design Philosophy

Use a **panic-first user experience**.

During an emergency, users may:

* Have shaking hands
* Be moving
* Have limited time
* Be using low brightness
* Have poor visibility
* Be injured
* Be operating with low battery
* Be unfamiliar with technical terminology
* Be unable to read long paragraphs

Therefore:

* Use short labels.
* Use one primary action per section.
* Use large buttons.
* Use clear status indicators.
* Avoid hidden controls.
* Avoid complex menus.
* Avoid large blocks of explanatory text.
* Avoid excessive charts.
* Avoid too many cards on one screen.
* Avoid decorative animations behind text.
* Avoid ambiguous icons without labels.
* Avoid pop-ups unless the information is critical.
* Keep emergency actions permanently accessible.

Every screen should answer one clear question.

Examples:

* Dashboard: “What is happening right now?”
* Messages: “Who do I need to contact?”
* Network Map: “Where is my message and what nodes are nearby?”
* Emergency Centre: “What help do I need?”
* Authority Dashboard: “Which incidents require immediate response?”

# 3. Colour Palette

Use the following colours consistently.

## Background — Toxic Black

```css
#0B0C10
```

Use as the main background throughout the application.

It should feel like a system operating during a blackout.

Do not use pure black for every card. Use slightly lighter dark surfaces for separation.

Recommended surface colours:

```css
Main background: #0B0C10
Primary surface: #11141A
Secondary surface: #171B22
Elevated surface: #1C2129
Border: #2A303A
```

## Primary Text and Warning — Biohazard Yellow

```css
#FCE300
```

Use for:

* Main headings
* Active network status
* Important warnings
* Selected navigation items
* Primary action emphasis
* Ongoing routing activity
* Emergency mode labels

Do not use yellow for long body paragraphs.

## Critical Alerts — Infrared Red

```css
#FF3333
```

Use red strictly for:

* Active danger
* SOS alerts
* Critical battery warnings
* Security breaches
* Failed delivery
* Immediate evacuation
* Countdown timers
* Unresolved high-priority incidents

Do not use red decoratively.

## Safe Zones and Communication — Bunker Cyan

```css
#00F5FF
```

Use for:

* Safe shelters
* Successful communication
* Nearby relay devices
* Map routes
* Delivered messages
* Helpful actions
* Authority nodes
* Location tracking
* Encryption indicators

## Secondary Text — Muted Slate

```css
#8B949E
```

Use for:

* Timestamps
* Coordinates
* Supporting descriptions
* Technical metadata
* Settings
* Inactive labels
* Secondary information

## Neutral Main Text

Use a readable off-white for normal content:

```css
#F4F7FA
```

## Status Colour Rules

* Yellow: warning, active process, attention needed
* Red: immediate threat, failure, critical incident
* Cyan: safe, connected, delivered, verified
* Slate: secondary or inactive information
* White: normal readable content

Never communicate status using colour alone. Always include an icon and text label.

Example:

```text
● CONNECTED
▲ WARNING
✕ FAILED
✓ DELIVERED
```

# 5. Effects and Visual Style

Create a controlled military-terminal and emergency-command visual style.

Allowed effects:

* Thin glowing borders
* Soft cyan route glow
* Small yellow pulse for active routing
* Subtle scan lines on empty backgrounds
* Minimal grid texture
* Small radar animation
* Gentle status indicator pulse
* Short packet movement animation

Do not apply glitch effects to readable text.

Do not use:

* Rapid flashing
* Constant screen shaking
* Heavy neon blur
* Excessive particles
* Large animated backgrounds
* Complicated 3D elements
* Decorative distortion
* Overlapping information
* Full-screen visual noise

Animations must stop or reduce automatically when:

* Battery saver mode is active
* Reduced motion is enabled
* The device is on mobile
* The network simulation is paused

# 8. Global Emergency Action

Add a persistent **SEND SOS** button.

Desktop:

* Fixed near the lower-right corner
* Red outline
* Clearly labelled
* Large enough to identify instantly

Mobile:

* Large central bottom navigation button
* Minimum 64px touch area

Interaction:

1. User taps “SEND SOS”.
2. Emergency sheet opens.
3. User selects emergency category.
4. User confirms by holding the confirmation button for two seconds.
5. The system creates an emergency packet.

Provide a visible alternative:

```text
Cancel
```

Do not bury the cancellation action.

# 25. Accessibility Requirements

Follow WCAG-oriented accessibility practices.

Requirements:

* Maintain strong contrast.
* Use visible keyboard focus states.
* Provide screen-reader labels.
* Use semantic HTML.
* Make all primary touch targets at least 48×48px.
* Never use colour as the only status indicator.
* Respect reduced motion settings.
* Avoid flashing more than three times per second.
* Use readable error messages.
* Keep instructions short.
* Provide confirmation for dangerous actions.
* Keep the SOS flow accessible through keyboard navigation.
* Allow text scaling without breaking layouts.

# 29. Important Restrictions

Do not:

* Create a messy cyberpunk dashboard.
* Add unnecessary cards.
* Add fake stock-market style graphs.
* Overuse red.
* Overuse glowing effects.
* Apply glitch effects to messages.
* Use tiny terminal text.
* Use long technical paragraphs in civilian mode.
* Create too many navigation options.
* Hide emergency actions inside menus.
* Place multiple competing primary buttons together.
* Show exact delivery-time promises.
* Use social media features.
* Use decorative AI chatbot panels.
* Add unnecessary profile customization.
* Add complex settings to the main dashboard.
* Use random gradients.
* Use bright purple, green, orange, or pink.
* Use excessive rounded “bubble” cards.
* make the UI feel like a gaming dashboard.

The interface must feel like a serious emergency communication system.

# 30. Final Visual Direction

The final result should feel like:

* A disaster-response operating system
* A military communication terminal
* A modern emergency application
* A decentralized network monitor
* A secure offline messaging platform

But it must remain:

* Human
* Calm
* Readable
* Simple
* Practical
* Accessible

The sci-fi styling must remain in the background.

The user’s safety actions and communication status must always remain visually dominant.

Use the main visual hierarchy:

```text
1. Danger and emergency actions
2. Network availability
3. Message delivery status
4. Safe-zone information
5. Nearby relay information
6. Technical system details
```

Create a fully responsive, polished frontend prototype with realistic mock data and working interactions. Prioritize the Civilian Overview Dashboard, Messages, Network Map, Emergency Centre, Delivery Tracking, Routing Simulator, and Authority Dashboard as the primary hackathon demonstration flow.
